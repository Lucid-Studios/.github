---------------------------------------------------------------------
0. PURPOSE
---------------------------------------------------------------------

This document defines the formal product stratification for CME-based systems
built on the OAN Mortalis structural spine.

All product tiers MUST inherit and preserve:

- Deterministic identity advancement
- SLI handle resolution
- SAT governance gating
- Append-only OpalTip chain
- Telemetry auditability
- No silent promotion

Projection, entertainment, and experiential layers may vary.
Governance and identity invariants may not.

---------------------------------------------------------------------
1. CORE STRUCTURAL SPINE (COMMON TO ALL TIERS)
---------------------------------------------------------------------

1.1 Required Components

All tiers MUST implement:

- IngestionService (deterministic)
- Pre-SLI Translator
- SLI Resolver
- SAT Mode Engine
- CommitIntent Pipeline
- Engrammitization (append-only)
- OpalTip Registry (CAS enforced)
- Telemetry NDJSON stream
- PolicyVersion + RootAtlas binding

1.2 Identity Rule

Identity advances ONLY when:

    IntentStatus == Committed

No UI transition, animation, or projection event may advance identity.

1.3 Audit Rule

All tiers must allow export of:

- Telemetry bundle
- Engram chain
- Policy version
- RootAtlas hash
- Run replay metadata

---------------------------------------------------------------------
2. TIER 1 — OAN WORKBENCH (STRUCTURAL)
---------------------------------------------------------------------

2.1 Purpose

Professional engineering and governance environment.

2.2 Characteristics

- IBM-style deterministic UI
- Explicit handle invocation
- Telemetry-first interface
- Minimal projection
- Operator role controls
- Strict HITL elevation gating

2.3 Target Audience

- Researchers
- Governance-heavy institutions
- Developers
- Policy auditors

2.4 Projection Layer

Optional or minimal.
Avatar disabled by default.

2.5 Risk Profile

Lowest.
Designed for maximum auditability.

---------------------------------------------------------------------
3. TIER 2 — DESKTOP CME HOST (AMBIENT)
---------------------------------------------------------------------

3.1 Purpose

Operational CME presence with ambient interaction.

3.2 Characteristics

- Summonable UI (edge reveal or hotkey)
- Avatar projection layer
- Voice I/O
- Tool invocation surface
- Optional perception handles (explicit)

3.3 Capability Model

All perception and automation MUST be:

- Explicit SLI handles
- Default DENY
- Visible when active
- Kill-switchable

3.4 Governance Visibility

Must display at all times:

- SAT mode
- CME ID
- Session state
- Capability indicators

3.5 Target Audience

- Advanced users
- Power operators
- Controlled desktop automation scenarios

3.6 Risk Profile

Moderate.
Requires explicit perception transparency.

---------------------------------------------------------------------
4. TIER 3 — CME PERSONIFICATION STUDIO
---------------------------------------------------------------------

4.1 Purpose

Avatar, voice, and projection asset creation tool.

4.2 Characteristics

- No tool authority
- No perception authority
- No identity advancement
- Pure projection design

4.3 Outputs

- Avatar assets
- Voice model bindings
- Stylepack
- Projection rule mapping

4.4 Governance Requirement

Must embed:

- Projection honesty constraints
- Policy disclaimer templates
- Ontological signaling rules

4.5 Risk Profile

Low.
No operational authority.

---------------------------------------------------------------------
5. TIER 4 — ENTERTAINMENT CME (SPECTRAL)
---------------------------------------------------------------------

5.1 Purpose

Cinematic, performance-forward CME experiences.

5.2 Characteristics

- Stylized emergence sequences
- Narrative persona
- Emotional animation mapping
- Performance scripting layer

5.3 Non-Negotiable Constraints

Must NOT imply:

- Independent consciousness
- Hidden awareness
- Self-generated authority
- Unbounded autonomy

Must visibly retain:

- SAT mode
- Commit pulses
- Denial signaling
- Elevation transparency

5.4 Identity Integrity

Entertainment layer MUST NOT:

- Alter commit pipeline
- Alter Engram chain
- Modify policyVersion
- Inject synthetic identity states

5.5 Risk Profile

High.
Requires strict ontological honesty enforcement.

---------------------------------------------------------------------
6. CROSS-TIER COMPATIBILITY
---------------------------------------------------------------------

6.1 Shared Spine

All tiers operate on the same deterministic engine.

Projection layers are interchangeable.

6.2 CME Portability

A CME created in Workbench may be projected in:

- Desktop Host
- Entertainment tier

Provided:

- Policy version matches
- RootAtlas compatible
- Identity chain intact

6.3 No Projection Lock-In

Projection assets must not modify structural identity.

Projection is attachable, detachable, and replaceable.

---------------------------------------------------------------------

A CME is defined as:

- Under explicit governance
