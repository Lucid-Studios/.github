# 🔐 Cryptic Store Refactor Snapshot

## Proposal: Move Cryptic Storage to SoulFrame, Keep Cryptic Root in CradleTek

### Status

Conceptual architectural shift under consideration.
Not yet implemented.
No schema changes committed.

---

# 1️⃣ Current Model (v0.1.0 Implicit)

Cryptic capability and storage both live in **CradleTek**.

CradleTek currently:

* Hosts GEL
* Hosts Gate
* Hosts RootAtlas
* Enforces SAT/HITL
* Applies Masking Policy
* Stores cryptic entries

This creates:

* System-level cryptic memory surface
* Shared cryptic partition risk
* Broader blast radius

---

# 2️⃣ Proposed Refactor

### Keep in CradleTek:

* Cryptic RootAtlas entries
* Cryptic eligibility rules
* SAT minimum enforcement
* Masking policy logic
* Gate evaluation discipline
* PolicyBundle pinning

CradleTek becomes:

> Constitutional authority over cryptic capability.

---

### Move to SoulFrame:

Per Operator / Bonded Agent cluster:

* CrypticStore (append-only)
* Cryptic index
* Cryptic symbol pointers
* Cryptic BraidIndex lineage
* Cryptic runtime tensor cache

Cryptic storage becomes:

> Identity-scoped and compartmentalized.

---

# 3️⃣ Architectural Effect

## Before:

```
CradleTek
 ├── GEL (Public/Private/Cryptic)
 └── Cryptic Root + Cryptic Store
```

## After:

```
CradleTek
 ├── GEL (Public/Private)
 ├── Cryptic Root (law only)
 └── Gate / SAT / Masking

SoulFrame (per identity cluster)
 ├── PublicStore (optional link to global GEL)
 ├── PrivateStore
 └── CrypticStore (local, append-only)
```

No layer violations introduced.

# 5️⃣ Security & Governance Gains

* Compartmentalization per bonded cluster
* Reduced cross-session leakage risk
* Reduced cryptic blast radius
* Drift containment per SoulFrame
* Clear separation of law (CradleTek) and memory (SoulFrame)

This aligns with:

> “Learn as many, act as one — without role bleed.”
